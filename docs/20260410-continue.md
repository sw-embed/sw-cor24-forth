### `\` and `(` comments don't work via `cor24-run -u` (HIGH PRIORITY)

**Symptom**: Feeding input containing `\` or `(` via `-u` causes the interpreter to hang. Only 3 RX bytes are consumed out of the full input. Subsequent commands on the same line are never reached.

```bash
# Works fine (no comments):
cor24-run --run forth.s -u '42 .\n99 .\n' --speed 0 -n 5M
# → "42  ok\n99  ok"  ✓

# Broken (backslash comment eats everything after it):
cor24-run --run forth.s -u '\ hello\n42 .\n' --speed 0 -n 5M
# → "42 " (no "ok", no "42 .")  ✗ — only 3 UART RX events, spins forever

# Broken (paren comment same issue):
cor24-run --run forth.s -u '( hello )\n42 .\n' --speed 0 -n 5M
# → "42 " (same hang)  ✗
```

Even with 50M instructions the emulator only delivers 3 bytes. This is not a instruction-budget problem.

**Root cause — emulator RX injection throttling**:

The emulator has two different byte-feeding mechanisms:

1. **Web UI** (`debugger.rs`): `feed_uart_byte()` is called once per tick (every `TICK_MS`). It checks RX status, and if the FIFO is empty, pops one byte from `uart_rx_queue` and calls `emulator.send_uart_byte()`. This gives the CPU time to process each byte between injections. **Works correctly.**

2. **CLI** (`cor24-run -u`): All input bytes are queued at startup. The emulator injects them into the 1-byte hardware FIFO at a throttled rate — approximately **1 byte per 10,000 instructions** (measured empirically: 100K instr → 10 bytes, 200K → 20, 500K → 50). **This is too slow for words that poll RX in tight loops.**

**Why `\` and `(` specifically fail**: These are IMMEDIATE words. The flow is:
1. `WORD` reads the token (e.g., `\` or `(`) from UART — consumes a few bytes
2. `INTERPRET` finds the word, sees IMMEDIATE flag, EXECUTEs it
3. The immediate word (`do_backslash` / `do_paren`) spins on `status bit 0` reading bytes until delimiter
4. But the emulator's throttled injection can't keep up — or stops entirely

The key mystery: with `AAAA...` (32 bytes), the emulator delivers all 32 given enough instructions (1 byte per 10K). But with `\ hello\n42 .\n` (15 bytes), even 50M instructions only delivers 3. Something about the execution pattern after `\` is found causes the emulator to stop injecting entirely. Possibly the emulator detects a tight spin loop at a single address and halts RX injection to avoid wasting time.

**TX (output) is fine**: `do_emit` correctly polls status bit 7 (TX busy) before writing. The emulator simulates TX busy realistically. All output works correctly in both CLI and web UI.

**Why existing tests pass**: The test suite doesn't use `\` or `(` in any `-u` input. It only tests these words indirectly (e.g., the "no WORDS leak" test calls WORDS which lists `\` and `(` in dictionary output, but doesn't execute them).

**What needs to happen to fix this** (one of):
- **Option A**: Fix `cor24-run` to inject RX bytes more aggressively (match web UI behavior of 1 byte per tick, not throttled by instruction count). This is an emulator change.
- **Option B**: Add an RX ring buffer in the Forth kernel itself, driven by the UART RX interrupt (`FF0010 IntEn`). On real hardware this is the correct approach — an ISR fires on each received byte and buffers it. The polling words (`KEY`, `WORD`) would read from the buffer instead of polling the hardware FIFO directly. This is a kernel change.
- **Option C**: Restructure `\` and `(` to not spin on UART directly. Instead, have INTERPRET handle them by calling WORD (which already has robust RX polling) and discarding the result. This avoids the tight spin loop but changes the comment semantics slightly.

### UART I/O reference (from macrolisp `io.h`)
```c
#define UART_DATA   0xFF0100   // write=TX, read=RX (auto-ack)
#define UART_STATUS 0xFF0101   // bit0=RX ready, bit1=CTS, bit7=TX busy

void putc_uart(int ch) {
    int tries = 50000;
    while ((*(char *)UART_STATUS & 0x80) && tries > 0) tries--;
    *(char *)UART_DATA = ch;
}

int getc_uart() {
    while (!(*(char *)UART_STATUS & 0x01)) {}
    return *(char *)UART_DATA;
}
```
